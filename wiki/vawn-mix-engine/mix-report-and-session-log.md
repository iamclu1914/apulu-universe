# Mix Report & Session Log

## Summary

The [[overview-and-architecture|Vawn Mix Engine]] generates two persistence artifacts: an HTML mix report (human-readable) and a JSON session log (machine-readable). Together they document every decision the system made and every measurement it took. This is part of Phase 5 (Reports + Polish), currently in progress.

## Mix Report (HTML)

Generated by `src/report_generator.py` using Jinja2 templates (`templates/report_template.html`). Available in HTML or Markdown format via config.

### Report Sections

1. **Session overview** — project name, stem count, total duration, processing time
2. **Per-stem analysis** — from `audio_analyzer.py`: peak dBFS, RMS dBFS, noise floor, clipping detection, hum detection, spectral centroid, crest factor
3. **Stage 1: RX Cleanup** — which preset was selected per vocal, why (noise floor level, clipping detected), files processed
4. **Stage 2: Nectar** — parameters applied per vocal track: compression ratio, reverb mix, formant shift, Auto-Level status, Unmask status
5. **Stage 3: Neutron** — parameters applied per instrument: EQ decisions, sidechain routing, parallel compression settings, gain offset applied
6. **Stage 4: TBC3 Reference** — frequency deviation analysis, bands that exceeded ±3 dB from reference, bus-level corrections made, number of correction passes
7. **Stage 5: Mastering** — Maximizer threshold, LUFS measurements per iteration, gain reduction, true peak verification, mono compatibility check
8. **Warnings and flags** — gain reduction exceeding `max_gain_reduction_db`, LUFS target not reached after 3 attempts, clipping detected post-processing
9. **Final output** — files exported, formats, paths

### Why This Matters

The report serves as a **decision transparency layer** — it answers "why did the system do that?" for every processing choice. This is analogous to how the [[apulu-prompt-generator/ui-modes-and-pipeline|Apulu pipeline]] streams stage-by-stage progress via SSE, but with much deeper detail.

For the artist, the report helps identify whether a suboptimal master is a mix problem (e.g., "808 caused 6.2 dB gain reduction — suggests remix") or a mastering problem.

## Session Log (JSON)

Persisted by `src/orchestrator.py` after each stage completes. Enables:

- **Resume after interruption** — if the pipeline crashes at Stage 3, restart from Stage 3 with prior state intact
- **Post-hoc inspection** — examine what each agent received and produced
- **Debugging** — trace parameter values across the entire pipeline

## Config Options

```yaml
output:
  report: true                # Generate mix report
  report_format: "html"       # "html" or "markdown"
```

## Key Takeaways

- The mix report is a **decision transparency layer** — documents why every processing choice was made
- The session log enables **pipeline resume** after interruption and **post-hoc debugging**
- Report flags problems the artist should address in the mix (e.g., excessive gain reduction suggests remix needed)
- Both artifacts together make the Mix Engine's decisions fully auditable
- This pattern (transparent decision logging) doesn't yet exist in the visual pipelines — noted as a gap in [[cross-topic/creative-pipelines-and-prompt-engineering|Creative Pipelines Comparison]]
- See [[overview-and-architecture]] for pipeline context, [[levels-and-gain-staging]] for the level targets referenced in the report
